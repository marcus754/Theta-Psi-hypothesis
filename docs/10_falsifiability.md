# 10. Фальсифицируемые критерии


## Канон времени

- Фундаментальный объект времени в гипотезе: тензор `Θ_{μν}` (см. `docs/11_derivation_path.md`).
- В формулах этого раздела используется редуцированная FRW-переменная
  `θ ≡ θ_eff`, где `θ_eff^2 = Θ_{μν} u^μ u^ν`, `u^μ u_μ = -1`.
- Все записи вида `n(θ)` в этом файле следует читать как `n(θ_eff, Θ, Ψ)`.
- Этот раздел описывает вычислительный редуцированный контур, а не полную
  тензорную динамику `Θ_{μν}`.

## Каноническая постановка (обязательно для всех разделов)

В текущем каноне первичны только две сущности: `Θ_{μν}` и `Ψ`.
Strong-field отклик задаётся напрямую составным инвариантом режима

```
J_refr(Θ,Ψ) = Φ_eff(θ_eff) · I_grad(Θ,Ψ),
I_grad(Θ,Ψ) = l_θ^2 · |∇_⊥θ|^2 + l_Ψ^2 · |∇_⊥Ψ|^2.
```

Рефрактивный сектор задаётся через

```
L_refr = M_*^4 F(J_refr),
n = n(θ_eff, Θ, Ψ),
ĝ_{μν} = g_{μν} + (n^2 - 1) u_μ u_ν.
```

Ключевое разделение для этого файла:
- `empirical` проверки опираются только на прямые data-facing observables и
  используются в `scripts/run_falsifiability_check.py`;
- `diagnostic` проверки могут использовать внутренние диагностики, scans и
  калибровочные артефакты, но не считаются жёстким критерием фальсификации.

## 0) Порядок работы

Фальсифицируемый контур здесь считается корректным только после того, как
отдельно закрыты четыре нижних шага:

1. строгий ковариантный вывод;
2. устойчивость и отсутствие ghost/лишних мод;
3. сжатие прогноза до 1-2 отличающих observable;
4. fit и сравнение с выбранными observational baselines только на уже закрытом каноне.

Пока эти шаги не закрыты, `diagnostic` сканы допустимы, но не должны
интерпретироваться как подтверждение гипотезы. Это принципиально важно:
хороший fit без первых трёх шагов не является физическим аргументом.

Цель: задать конечный набор численных критериев, при нарушении которых версия
гипотезы на базе `Θ_{μν}, Ψ` считается опровергнутой в текущем приближении.

## 1) Пакет слабопольных тестов

Критерии проверяются через `scripts/run_prediction_suite.py` и
`scripts/run_falsifiability_check.py`.

- Меркурий (аномальная прецессия):
  - целевое значение `42.98"/век`,
  - допустимое отклонение: `|pull| <= 3σ`.
- Красное смещение на поверхности Солнца:
  - диапазон `1.8e-6 .. 2.4e-6`.
- Шапиро (roundtrip, near conjunction):
  - диапазон `1.0e-4 .. 4.0e-4` секунд.
- Отклонение света у лимба Солнца:
  - диапазон `1.70 .. 1.80` угл. сек.

## 2) Критерий по совместной области

По `scripts/run_joint_region_scan.py`:

- должна существовать ненулевая 3σ-область (`N_3sigma >= 1`),
- при этом 3σ-область не должна занимать почти всю сетку
  (`f_3sigma <= 0.20`) — иначе модель нефальсифицируема на данном наборе тестов.

## 3) Diagnostic strong-field scans

Сканы вроде `scripts/run_compact_constraint_scan.py` и inferred-переходы из
`strong_field_bounds_registry.json` остаются в репозитории только как
архивная диагностика. Они могут использовать inferred-переходы вида
`наблюдение -> inferred индикатор`, поэтому не входят в PASS/FAIL по falsifiability.

## 4) Машиночитаемая спецификация

Пороговые значения для empirical falsifiability вынесены в:

- `fitting/data/prediction_targets.json`.

Автоматический чек:

- `python scripts/run_falsifiability_check.py`.

Для текущей версии канона приоритетны два класса прямых отличающих тестов:

1. strong-field saturation tests на первичном compact-канале
   (`z_surface`, `delay`, `ok_no_horizon`), а `ring_uas` и
   `echo_delay_ms` использовать как downstream forward checks;
2. background compatibility tests на `CC+SN+BAO`, но только как второй слой
   после закрытия пунктов 1-3 из раздела выше.

Если итог `FAIL`, версия считается незавершенной и требует доработки по
empirical-контуру. Diagnostic-отчёты могут падать или расходиться без
автоматического статуса фальсификации.
